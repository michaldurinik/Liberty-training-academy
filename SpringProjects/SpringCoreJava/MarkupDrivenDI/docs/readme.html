Basic Spring Dependency Injection

This project illustrates the basic Spring syntax for dependency injection. In the package demos.v1 we have some code that contains encapsulated dependencies. The packages demos.v2 to demos.v10 externalize the dependencies in various ways. The table below describes the techniques used in each package.

Package NameConfiguration FileDescription
demos.v1 N/A The Program class creates and uses a Shop object, which in turn creates and uses a PricingEngine, PaymentEngine and StockCheckEngine. It is not possible to use or test the shop without its three dependencies, and the dependencies are all on concrete types.
demos.v2 config/versionTwo.xml We use Spring to introduce dependency injection by:
  • Splitting each dependency into an interface and basic implementation
  • Injecting the dependencies into the Shop via constructor arguments
  • Creating a Spring 'wiring' file to manage the dependencies
  • In the main method of Tester using Spring to build the shop
demos.v3 config/versionThree.xml This is identical to demos.v2 except that we use properties for dependency injection rather than constructor arguments.
demos.v4 config/versionFour.xml This is identical to demos.v3, except that we nest the bean declarations in the configuration file.
demos.v5 config/versionFive.xml This example illustrates how Spring can cope with existing static factory methods. Assume we have the class PaymentEngineImpl, instances of which can only be created via the static method instance. The class PaymentImplTest shows how to build and use an instance of this class via the factory-method attribute of the bean element.
demos.v6 config/versionSix.xml This example is similar to demos.v5, except that a PaymentEngineImpl object can only be created by first creating a PaymentEngineFactory and calling its instance method. This shows that Spring supports factory classes as well as factory methods.
demos.v7 config/versionSeven.xml This example is identical to previous ones, except that we use autowiring by type instead of explicitly specifying dependencies.
demos.v8 config/versionEight.xml This example is identical to demos.v7, except that we use autowiring by name.
demos.v9 config/versionNine.xml This example shows how Spring lets you create and inject collections of values. We wish to extend our PricingEngineMock and StockCheckEngineMock classes so that, instead of returning dummy values, they use test values from collections injected by Spring.
demos.v10 config/versionTen.xml This example shows how one Spring bean declaration can inherit from another. The standard shopWithMocks bean is now declared as abstract and used as the basis for testShop1, testShop2 and testShop3.
demos.v11 config/versionEleven.xml Shows how Spring Expression Language (Spring EL) can be used to simplify the bean configuration file.
